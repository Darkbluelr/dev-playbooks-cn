# 提案对辩工作流（Proposal Debate）

目标：在 proposal 阶段通过“提案-质疑-裁决”三角对抗，避免共识稀释与机械指标误导。

目录约定（协议无关）：
- 变更包：`<change-root>/<change-id>/`
- 提案：`<change-root>/<change-id>/proposal.md`

强规则：
- Proposal Author / Challenger / Judge 必须独立对话/独立实例。
- Challenger 只能基于已落盘的 `proposal.md` 与设计资料提出异议，不得“私下共创”。
- Judge 必须给出明确裁决，不得折中含糊。

流程：
1) **Proposal Author**：使用 `devbooks-proposal-author` Skill 产出 `proposal.md`（含 Debate Packet）。
2) **Proposal Challenger**：使用 `devbooks-proposal-challenger` Skill 输出质疑报告。
3) **Proposal Judge**：使用 `devbooks-proposal-judge` Skill 输出裁决报告。
4) **写回提案**：将裁决结论写回 `proposal.md` 的 Decision Log。
5) **进入后续产物**：Design/Spec/Plan 仅在裁决为 Approved 或 Revise 后启动。

验收口径：
- `proposal.md` 中必须包含 `Decision Log`，且裁决状态为 `Approved | Revise | Rejected`。
- 若为 `Revise`，必须列出“必须修改项”并完成后重新裁决。
- 可选自动检查：`proposal-debate-check.sh <change-id>`（脚本位于本 Skill 的 `scripts/proposal-debate-check.sh`）
